Ibex is a 32-bit in-order \ac{RISC-V} core used across the lowRISC ecosystem, most notably in OpenTitan. It uses a 2-stage baseline pipeline with an optional writeback stage and exposes separate instruction and data memory ports through a request/grant/response handshake. Ibex supports either RV32I or RV32E and can be configured with the C, M, and B extensions, with selectable implementation variants such as multiplier style and B profiles (balanced, OTEarlGrey, or full). It also exposes several microarchitectural options that affect area, performance, and \ac{FT}, including a choice between prefetch buffer and instruction cache, multiple register-file implementations (\ac{FF}-based, \ac{FPGA}-oriented, or latch-based), and optional branch optimizations like a branch target \ac{ALU} and experimental static prediction for taken branches.

Ibex implements the standard \ac{RISC-V} \ac{CSR} set, which in this work was mainly used for performance monitoring and communication with the firmware. The cycle counter \texttt{mcycle} (\ac{CSR} address \texttt{0xB00}) and the retired instruction counter \texttt{minstret} (\ac{CSR} address \texttt{0xB02}) are always available and 64-bit wide. Optional event counters \texttt{mhpmcounterX} (\texttt{0xB03} and above) and \texttt{mhpmcounterXh} (\texttt{0xB83} and above) can also be implemented, with a configurable number of counters and counter width at synthesis time. Counters can be individually enabled or disabled through \texttt{mcountinhibit} (\ac{CSR} address \texttt{0x320}): bit 0 controls \texttt{mcycle}, bit 2 controls \texttt{minstret}, and bit X controls \texttt{mhpmcounterX}. Reads are non-destructive, so values can be sampled at benchmark boundaries and subtracted to obtain per-run metrics. While developing \ac{Fatori-V}, to validate the \ac{CSR} read system, \texttt{marchid} (a static \ac{CSR} with address \texttt{0xF12}) was read and compared against its documented fixed value \texttt{0x0000\_0016}. \Cref{tab:ibex_perf_csrs_example} displays the most relevant \acp{CSR} for this work.

\section{Ibex and Fault Tolerance}
\label{chap5:ibex_ft}

Ibex was selected partly because it includes native \ac{FT}-related support. This section introduces the native Ibex \acp{FTM} and detail native \ac{FT} features that are relevant for \ac{Fatori-V}'s development.

The native SecureIbex configuration enables Ibex' \acp{FTM}, mostly already introduced in \Cref{chap2:ftms}. Dual-Core Lockstep instantiates a shadow core, delays the inputs and outputs used for comparison, and raises an internal major alert when the compared outputs diverge. Shadow \acp{CSR} protect selected control registers by keeping complemented copies and checking consistency. Register-file \ac{ECC} widens register-file data with check bits and raises an internal major alert when corruption is detected. ICache \ac{ECC} protects cache tag and data banks and raises a minor alert on detected errors.

Other native Ibex checks follow the same checksum principle. Bus integrity checking verifies instruction and data responses against integrity bits and raises a bus major alert on mismatch. Hardened \ac{PC} checks whether sequential execution advances the \ac{PC} by the expected instruction length, except after branches and reset. The register-file write-enable and read-address glitch checks protect register-file control signals against invalid changes. Secure Guards use multi-bit encodings and guarded control paths for selected control signals, reducing the chance that a single-bit fault turns into a valid control value.

Data Independent Timing removes selected data-dependent timing paths, such as branch timing differences and early completion in multiply or divide cases. Dummy Instructions insert random operations that do not change the program result. These mechanisms are not used for \ac{SEU}-related \ac{FT}. Their purpose is to reduce timing and power predictability and make external targetted attacks harder to perform.

Ibex does not provide complete error handling hardware by default, but it exposes control and status ports that can be used by integration logic to detect faults and trigger recovery. These ports, listed in \Cref{tab:ibex_error_ports}, include three alert outputs, a multi-bit fetch-enable input that can stop instruction fetching, a sleep indicator that reports core activity, and a double-fault flag.

However, Ibex only exposes these ports. It does not aggregate errors across the \ac{SoC}, count error occurrences, or compute any \ac{FT}-related. Furthermore, alert outputs are status signals, not counters, meaning that they remain asserted until cleared. Counting detections, extracting quantitative \ac{FT} metrics, and reacting to reported errors requires the implementation of a fault manager.

\section{System-on-Chip Integration}
\label{chap5:soc-integration}

The main reason behind abandoning the VeeRwolf EL2's path in \Cref{chap4:summary} was the unexpected implementation complexity that arose during development. This time, the locally developed IOb-\ac{SoC} system was chosen to improve support and mitigate integration constraints later. The integration process resulted in IOb-\ac{SoC}-Ibex, a \ac{SoC} plus \ac{CPU} system analogous to IOb-\ac{SoC}, that rely on the Py2HWSW python framework for hardware generation and building. Both the python platform and the actual integration process are detailed next in \Cref{chap5:py2hwsw,chap5:iob-soc-ibex}.





% -------------------------------------------------------------
\subsection{Py2HWSW Infrastructure}
\label{chap5:py2hwsw}

Py2HWSW is a Python-based framework for embedded hardware and software design. The user describes a system through a small set of \texttt{.py} entry files that declare modules, connections, and configuration parameters, and the framework generates a complete project workspace with the \ac{RTL}, software support, scripts, and Makefiles required for emulation, simulation, synthesis, and \ac{FPGA} bitstream generation. The workflow is organized around three directories, the Setup Directory, the Py2HWSW Directory, and the Build Directory. IOb-\ac{SoC} is an example of a system built with this model.

The Setup Directory is the user repository and the place where development changes are applied. It contains the Python system description, project-specific hardware and software sources, and any overrides to the default infrastructure. The Py2HWSW Directory contains the framework core. It provides the reusable module library, templates, scripts, and build infrastructure used as defaults when the main directory does not provide a local implementation. The Build Directory is the generated output workspace, created outside the main repository. It merges the contents of the Setup Directory and its declared submodules with the defaults from the Py2HWSW library, producing a complete set of sources and Makefiles for the supported flows, and it can be deleted and regenerated when needed. Both IOb-\ac{SoC} and IOb-\ac{SoC}-Ibex repositories are Setup Directories that rely on Py2HWSW for building and execution.



% -------------------------------------------------------------
\subsection{IOb-SoC-Ibex}
\label{chap5:iob-soc-ibex}

IOb-SoC-Ibex, developed in this work, is analogous in structure to IOb-SoC, but is built from scratch to host Ibex as the \ac{CPU}. The Ibex core is provided through a separate wrapper repository, IOb-Ibex \cite{iob-ibex}, also developed in this work, which includes the official Ibex repositories as submodules and provides the hardware integration required by the Py2HWSW structure. To achieve a functioning system, two classes of modifications were required: build workflow integration (to generate and synchronize native Ibex sources within the Py2HWSW structure) and hardware interfacing (to connect the Ibex memory handshake to the \ac{SoC} bus). The build workflow was organized into three layers, visible in \Cref{fig:ibex+iob}, each added as a Git submodule of the upper one. 

The top layer, IOb-SoC-Ibex , acts as the \ac{SoC} project and is forked from IOb-SoC. At this level, the first step was selecting Ibex as the \ac{CPU} in \texttt{iob\_soc.py} with \texttt{"cpu": "iob\_ibex"}, so that the Py2HWSW flow searches for \texttt{iob\_ibex.py} in the project's submodules. Since this layer is meant to be controlled by \ac{Fatori-V}, two hook mechanisms were added to controll the Vivado TCL flow and benchmark compiling. For the Vivado side, the main \texttt{build.tcl} script was modified to call \texttt{source\_probe} at fixed stages of execution (pre synthesis, post opt, post route, pre bitstream, and post bitstream). If the user provides the Makefile argument \texttt{VIVADO\_INPUT} pointing to a directory containing any of these files, each one is sourced automatically at the corresponding point of the Vivado flow. On the software side, the default firmware flow was extended so that a benchmark can be selected from an external directory through the Makefile argument \texttt{BENCHMARK\_DIR}. If this argument is provided, the build system compiles the benchmark sources from that directory instead of the default Hello World program, and the firmware wrapper calls a \texttt{main} function defined in \texttt{benchmark.h}, a mandatory file for any benchmark integrated in this system. Together, these two hooks made the top layer externally configurable without requiring permanent edits to the base \ac{SoC} sources for each experiment. Since Ibex relies on SystemVerilog packages, and the Py2HWSW is designed only for Verilog source files, the default \ac{FPGA} build flow's compilation ordering was changed so that \texttt{*\_pkg.sv} files are compiled before other source files that import them. This change was implemented in the \ac{FPGA} Makefile, as shown in \Cref{lst:src_order}.


\begin{lstlisting}[float=h,language=matlabfloz,caption={Portion of the modified Py2HWSW \ac{FPGA} Makefile enforcing package ordering}, label={lst:src_order}]
VHDR+=$(wildcard ../src/*.vh) $(wildcard ./src/*.vh)
PCKS+=$(wildcard ../src/*_pkg.sv)
VSRC+=$(PCKS)
SVRC+=$(wildcard ../src/*.sv)
VSRC+=$(filter-out $(PCKS), $(SVRC))
VSRC+=$(wildcard ../src/*.v) $(wildcard ./src/*.v)
\end{lstlisting}

IOb-Ibex, the second layer, is responsible for exposing the Ibex \ac{RTL} to the Py2HWSW Build Directory. Since Py2HWSW copies whatever exists under \texttt{iob\_ibex/hardware/src/} into the build tree, IOb-Ibex keeps only locally modified or integration-specific modules there, and avoids storing the full native Ibex source tree. The remaining untouched Ibex sources are instead generated and copied (by new Makefile targets) from the official repositories (third layer) directly into the Build Directory, excluding files that already exist in \texttt{hardware/src/} so that local fixes remain in effect. Because Ibex is SystemVerilog-based, \texttt{iob\_ibex.py} disables Py2HWSW's automatic HDL generation through \texttt{"generate\_hw": False}, and the wrapper is provided manually as \texttt{iob\_ibex.sv}. The Python description is still required so Py2HWSW can parse interfaces and derive system parameters such as clock and \ac{UART} settings.

The third layer contains the official Ibex repository, included as a submodule of IOb-Ibex and used as the source for those generation targets. Its FuseSoC and \texttt{make}-based export flows are invoked by IOb-Ibex to produce a complete set of SystemVerilog sources and copy them into the Py2HWSW Build Directory. Py2HWSW projects use Nix environments to manage dependencies, but the packages required by the Ibex export flow were not compatible with that setup. The adopted solution was to include Ibex-Demo-System \cite{ibex-demo-system} as an additional submodule and reuse its Nix Flakes environment to run the required FuseSoC targets inside a compatible shell.

\subsection{AXI Bridge for Ibex}
\label{chap5:ibex2iob}

IOb-SoC connects the \ac{CPU} to memory and peripherals through an \ac{AXI} bus, while Ibex uses a simpler request, grant, and response handshake on separate instruction and data ports. To bridge these interfaces, a dedicated AXI-to-Ibex hardware bridge was developed and instantiated in the \ac{CPU} wrapper. \Cref{fig:ibex_protocol} illustrates the native Ibex LSU protocol, where a request stays asserted until granted, and completion is signalled by a one-cycle response pulse that may include data and an error flag.

\begin{figure}[h]
\centering
    \includegraphics[width=0.9\textwidth]{./Images/Ibex Protocol.png}
    \caption{Ibex LSU interface timing diagram. Ibex' perspective.}
    \label{fig:ibex_protocol}
\end{figure}

The developed module, \texttt{iob\_ibex2axi} shown in \Cref{fig:ibex+iob}, provides Ibex-facing ports on one side and an \ac{AXI} interface toward IOb-SoC on the other. In this version, the adapter targets the subset of signals needed for a functioning \ac{SoC} integration, translating the core read and write transactions into an \ac{AXI}-Lite style behaviour. Advanced \ac{AXI} features such as burst transfers are not implemented, and unused attributes are tied to default values. The Ibex side of the interface and the \ac{AXI} side signals covered by the adapter are summarized in \Cref{tab:ibex2axi}.

A transaction starts when Ibex raises \texttt{ibex\_*\_req\_i}, informing that it is requesting memory access. Alongside the request, Ibex provides \texttt{ibex\_*\_addr\_i} with the address and, in store transactions, \texttt{ibex\_data\_we\_i} and \texttt{ibex\_data\_wdata\_i} (the byte enable and write data). The bridge translates the transaction into the corresponding \ac{AXI} signals by driving \texttt{*bus\_arvalid\_o} for reads or \texttt{dbus\_awvalid\_o}
/\texttt{dbus\_wvalid\_o} for writes, then asserts \texttt{ibex\_*\_gnt\_o} once the \ac{AXI} slave is ready to accept the transfer. After the transaction is granted and the data being validated through \texttt{*bus\_rvalid\_i} or \texttt{dbus\_bvalid\_i} (reads or writes), the bridge sends a one-cycle \texttt{ibex\_*\_rvalid\_o} pulse to Ibex, forwards read data on \texttt{ibex\_*\_rdata\_o}, and asserts \texttt{ibex\_*\_err\_o} when the \ac{AXI} response indicates an error.

The hardware bridge implements the signals described in \Cref{tab:ibex2axi} through direct combinational assignments. Since the Ibex data interface uses \texttt{ibex\_data\_we\_i} to distinguish loads from stores, the same request signal is routed either to the \ac{AXI} read address channel, $\texttt{dbus\_arvalid\_o} = \texttt{ibex\_data\_req\_i} \; \& \; \sim\texttt{ibex\_data\_we\_i}$, or to the write address and write data channels, $\texttt{dbus\_awvalid\_o} = \texttt{dbus\_wvalid\_o} = \texttt{ibex\_data\_req\_i} \; \& \; \texttt{ibex\_data\_we\_i}$. The address signals \texttt{dbus\_awaddr\_o} and \texttt{dbus\_araddr\_o} are driven from \texttt{ibex\_data\_addr\_i}, while \texttt{dbus\_wdata\_o} and \texttt{dbus\_wstrb\_o} are driven from \texttt{ibex\_data\_wdata\_i} and \texttt{ibex\_data\_be\_i}, simply forwarding the data and address signals.  The response valid signal is computed as $\texttt{ibex\_data\_rvalid\_o} = \texttt{dbus\_rvalid\_i} \; || \; \texttt{dbus\_bvalid\_i}$, because Ibex expects one response indication per request, while \ac{AXI} reports load completion through the read data channel and store completion through the write response channel. The response channels are fixed through \texttt{dbus\_bready\_o=1}, \texttt{dbus\_rready\_o=1}, and \texttt{ibus\_rready\_o=1}, since Ibex does not provide a separate response interface. Unused \ac{AXI} fields are tied to default values, including \texttt{*\_len\_o=0}, \texttt{*\_burst\_o=0}, and \texttt{dbus\_wlast\_o=1}, which restricts this bridge to single transfers.
