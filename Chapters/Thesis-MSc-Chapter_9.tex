\section{Experimental Setup}
\label{chap9:experimental_setup}

This section presents the experiments made using the \ac{Fatori-V} platform. Each experiment is fully defined with a single \texttt{.yaml} file, which describes the target hardware configuration (architecture options and enabled \acp{FTM}), software benchmarks, and fault-injection campaign parameters. For each run, \ac{Fatori-V} automatically builds the \ac{FPGA} image, programs the board, launches the required host-side consoles, executes the benchmark suite, and produces run directories containing logs and results tables. The software benchmarks include a Hello World program, new Fatori Stress software, and two standard benchmarks, CoreMark and Embench-IoT. All experiments can be reproduced by using the main \ac{Fatori-V} GitHub repository in \cite{FatoriV}, using the \texttt{python3 fatori-v.py} command.

\begin{figure}[h]
  \centering
  \includegraphics[width=0.9\linewidth]{Images/chap9_exp_setup.png}
  \caption{Fatori-V system used in the experimental setup.}
  \label{fig:chap9_exp_setup}
\end{figure}

\Cref{fig:chap9_exp_setup} shows the system used in the experiments. The \texttt{fatori-v.py} script runs all \texttt{.yaml} tests. Each test synthesizes and implements the FPGA hardware, programs the FPGA, configures the \ac{FI}, and waits for the firmware to finish execution. Next, the text file produced by the firmware, containing the measured results, is analyzed to generate the result tables.

%orange
All \acp{FTM} implemented in this work report their error signals to the central \texttt{fault manager} (orange paths in \Cref{fig:chap9_exp_setup}). The fault manager writes the error data to the implemented Ibex \acp{CSR}.

%blue
The firmware reads the \acp{CSR} and sends the metrics to the user desktop console (blue path in \Cref{fig:chap9_exp_setup}). The \ac{Fatori-V} platform parses the results and consolidates them into the tables used in the analyses presented below.

%green
During execution, faults are injected (green paths in \Cref{fig:chap9_exp_setup}) using the \ac{FI} \ac{UART}, and decoded either as "R" commands (register injections), or \ac{SEM} commands (configuration-memory injections).

\section{Fatori-V Validation} 
\label{chap9:fatori_validation} 

Before evaluating the \acp{FTM}, a set of experiments was performed to validate \ac{Fatori-V} itself. These experiments checked whether configuration memory injections reached the intended physical regions and whether the implemented time profiles followed their configured behaviour.

The area precision was evaluated through the Pblock Model. For each pblock target, the estimated target size was compared with the real required size later reported by Vivado. The normalized size error was computed as:

\[
E_{\mathrm{pblock}} = 100 \cdot \frac{|S_{\mathrm{real}} - S_{\mathrm{est}}|}{S_{\mathrm{est}}}
\]

The corresponding accuracy was computed as:

\[
A_{\mathrm{pblock}} = 100 - E_{\mathrm{pblock}}
\]

Targets with configurable parameters, such as the \ac{ALU} and the Controller, were evaluated in the baseline configuration and in two additional configurations with more enabled options. Targets without configurable parameters, such as the Writeback Stage, the Prefetch Buffer, and the Branch Predictor, were evaluated once. The Pblock Model accuracy results are shown in \Cref{fig:chap9-pblock-model-accuracy}.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.9\textwidth]{Images/chap9_pblock_model_accuracy.png}
    \caption{Pblock Model accuracy across the evaluated pblock targets.}
    \label{fig:chap9-pblock-model-accuracy}
\end{figure} 

The Pblock Model reached 100\% accuracy for non-configurable targets, such as the Branch Predictor and the Prefetch Buffer, 85\% for the most complex configurations of parameterizable targets, such as the ID Stage and the \ac{LSU}, and 91.63\% on average.

The time profiles were evaluated by running timestamped \ac{FI} campaigns for each implemented profile. Each campaign was configured to produce approximately 12000 injections. The measured timestamps were used to reconstruct the observed profile and compare it with the ideal profile generated from the same configuration. The Uniform, Poisson, and \ac{MMPP2} profiles were evaluated as inter-injection period distributions. The Ramp and Microburst profiles were evaluated by grouping timestamps into time bins and comparing the measured injection rate in each bin with the configured rate.

For each profile, the measured and ideal profiles were compared using the same bins. The profile error was computed from the normalized absolute difference between the measured value \(y_i\) and the ideal value \(y^{\mathrm{ideal}}_i\) in each bin:

\[
E_{\mathrm{profile}} =
\frac{\sum_{i=1}^{n} |y_i - y^{\mathrm{ideal}}_i|}
{\sum_{i=1}^{n} y^{\mathrm{ideal}}_i}
\]

The profile match score was computed as:

\[
\mathrm{Profile\ Match} = 100 \cdot (1 - E_{\mathrm{profile}})
\]

A higher profile match means that the measured profile follows the configured profile more closely.

\section{Comparison of Fault Tolerance Mechanisms}
\label{chap9:general-ftm-testing}

This section compares the evaluated \acp{FTM} under the same \ac{FI} conditions. Each hardware setup contains a defined set of enabled \acp{FTM}. In the individual \ac{FTM} experiments, only the mechanism under test is enabled. The comparison also includes a Baseline with no \acp{FTM}, SecureIbex with the native Ibex \acp{FTM} enabled, and one setup with all evaluated \acp{FTM} enabled.

The native Ibex \acp{FTM} evaluated individually are Lockstep, Dummy Instructions, Data Independent Timing, Hardened \ac{PC}, PMP, RegFile \ac{ECC}, RegFile Glitch Checks, Secure Guards, and Shadow \acp{CSR}. RegFile Glitch Checks groups the RegFile RADDR and \ac{RF} WE glitch checks. The two \acp{FTM} implemented for \ac{Fatori-V} are Register Redundancy and Logic Redundancy.

Register Redundancy was enabled for all 197 wrapped registers, with \(N=3\), \(M=2\), and Correction enabled. Logic Redundancy was enabled for the five supported logic targets listed in \Cref{chap6:mon}, namely the \ac{ALU}, \ac{LSU}, Controller, Decoder, and IF Stage, with \(N=3\) and \(M=2\).

The Baseline contains no \acp{FTM}, but keeps the Fault Manager and monitoring hardware enabled because these blocks are required to collect \ac{FT} data. It uses the Branch Predictor and Branch Target \ac{ALU}, and enables the M, C, and B \ac{ISA} extensions. The multiplier is the native Ibex Single Cycle multiplier, and the bit manipulation unit is the native Ibex Balanced implementation. All experiments use a \ac{CPU} frequency of 60 MHz.

For each hardware setup, \ac{Fatori-V} ran the same set of \ac{FI} campaigns. Each setup was tested with seven injection rates, three register/configuration memory ratios, and two injection scopes, producing 42 campaigns per setup and 588 experiments overall. The same seeds, benchmarks, \ac{FI} configuration, and target-pool replay were used across setups, so each setup followed the same injection targets and timing.

All campaigns use the Poisson time profile, a fixed duration of 120 s, and injection rates of 1 Hz, 10 Hz, 50 Hz, 100 Hz, 500 Hz, 1000 Hz, and 5000 Hz. The register/configuration memory ratio defines how injections are split between wrapped registers and configuration memory. A ratio of 0 injects only configuration memory faults, a ratio of 1 injects only register faults, and a ratio of 0.0001 combines both fault types. This mixed ratio approximates the proportion between the 197 wrapped registers and the 1837420 \ac{EBD} configuration addresses exported by Vivado for the Baseline, which represent the possible configuration memory injection locations. The two injection scopes are the full device and the selected Pblock targets.

The same seeds, benchmarks, and \ac{FI} configurations were used across experiments. The \ac{FI} target pool from the first experiment was reused in the remaining experiments, so each campaign followed the same injection targets and timing. The executed benchmarks were Fatori Stress with 100000 iterations and all targets enabled, CoreMark with 1000 iterations, and the Embench-IoT benchmarks crc32, cubic, edn, huffbench, matmult\_int, md5sum, minver, nbody, nettle\_aes, and nettle\_sha256. Performance was measured from CoreMark and Embench-IoT. Experiment success was measured with the Fatori Stress checksum check.

\Cref{tab:results_wd_all_ratios} shows that every configuration without Register Redundancy crashes when register faults are injected. With Register Redundancy, the system passes every register-only injection rate. This shows that Register Redundancy is able to protect the \ac{SoC} against \ac{FF}/register corruption, while the other evaluated mechanisms do not prevent crashes under the same fault type.

In configuration-memory-only campaigns, the implemented Logic Redundancy \ac{FTM} is also able to improve the results. With whole-device injection, the configurations without Logic Redundancy pass up to 100 Hz and fail at 500 Hz. With Logic Redundancy, the benchmark passes up to 500 Hz and fails at 1000 Hz.

Mixed campaigns are dominated by configuration memory injections due to the selected ratio. However, the experiments with only Logic Redundancy perform worse than the experiments with Register Redundancy. The way that that the \ac{FI} tool is implemented, the mixed target pool can still include a register fault before converging to the configured ratio average. In these campaigns, the first mixed injection is a register injection. This was confirmed by the per-type fault counts exported by \ac{Fatori-V}, and is also visible in the Observability values. 

In the Register Redundancy rows, 0.8333\% at 1 Hz corresponds to \(1/120\) (one register injection, 120 total injections), and 0.0833\% at 10 Hz corresponds to \(1/1200\) (one register injection, 1200 total injections). As a result, configurations without Register Redundancy still crash even at the lowest mixed rate.

The native Ibex \acp{FTM} implementations did not increase the \ac{FT} of the \ac{SoC}. Data Independent Timing and Dummy Instructions do not detect or correct corrupted values, as predicted in \Cref{chap5:ibex_ft}, because they are not designed for \ac{SEU}-like corruption. 

Most native \acp{FTM} only cover very specific and narrow parts of the core. Hardened \ac{PC}, PMP, RegFile Glitch Checks, and Secure Guards protect specific control, address, register-file, or protocol conditions. These mechanisms can detect \acp{SEU} when corruption reaches the specific core structures they monitor. However, their limited coverage makes them more suitable for protecting relevant core regions against targeted external attacks than for broad random-fault campaigns.

Lockstep and Shadow \acp{CSR} provide broader detection, but they do not implement correction. Both mechanisms raise major errors, so the faults they are able to detect are also the faults most likely to make the system crash. A possible improvement would be to add recovery after detection, such as checkpoint rollback, local state repair, or voting with more than two replicas.

Whole-device campaigns show how the implemented \ac{SoC} behaves when faults can affect any configuration memory location. This gives a system-level view of whether the enabled \acp{FTM} are enough for broad fault exposure. 

However, the whole-device scope does not always test each \ac{FTM} within the region it is meant to protect. In ratio 0 campaigns, the benchmark still passes several rates without Logic Redundancy, while Observability remains low. This indicates that many injections either land outside the protected logic targets or affect regions that are not exercised by the benchmark. To study configuration memory faults in more relevant core regions, the same experiments were repeated with the injection scope limited to the Pblock targets described in \Cref{chap7:pblock}. This scope depends on the Pblock Model evaluated in \Cref{chap9:fatori_validation}, because the \ac{FI} tool uses the generated Pblock regions to select configuration addresses for each target module.

Comparing \Cref{tab:results_wd_all_ratios,tab:results_modules_all_ratios} shows that the Pblock-target scope makes configuration memory faults affect the benchmark earlier. Without Logic Redundancy, the first failing rate moves from 500 Hz to 50 Hz. This indicates that the selected Pblock targets include regions that are more relevant to the executed workload than a random whole-device injection scope.

Logic Redundancy still improves the result under the Pblock-target scope. With Logic Redundancy, the first failing rate moves from 1000 Hz to 500 Hz, instead of from 500 Hz to 50 Hz. Observability is also higher than in the whole-device scope for the same ratio 0 campaigns. This supports the conclusion that Logic Redundancy is useful when configuration memory faults reach the protected logic, and that extending it to more modules would increase its coverage.

The difference between the two scopes also explains the lower Observability values in the whole-device campaigns. Many configuration memory injections land outside the protected logic targets, or in regions that do not affect the executed benchmark. In those cases, the benchmark may still pass while no detection is counted.

For ratio 1 with Register Redundancy enabled, for both scopes, the system passes every tested rate, but Observability remains close to 26.5\% instead of 100\%. Since register-only injections are expected to be detected and corrected by Register Redundancy, this points to the aggregation and counting path. Several detections can be merged before reaching the final counter. Observability therefore also exposes limitations in the monitoring path, not only the behaviour of the \acp{FTM}.

For ratio 0, Observability remains low and decreases as the injection rate increases. This reflects two limits of the current Logic Redundancy implementation. Only five logic targets can detect configuration memory faults, and detected configuration memory faults are not repaired. Corrupted configuration bits can therefore remain in the design and contribute to later failures.



\subsection{Performance and Hardware Resources}

The experiments in \Cref{tab:results_wd_all_ratios,tab:results_modules_all_ratios} showed no performance differences between the evaluated \acp{FTM}. All valid runs reported \ac{IPC}=0.28, CoreMark score 44, and Embench-IoT score 0.076. These results show that the tested \acp{FTM} did not change the measured benchmark performance.

Native Ibex \acp{FTM} did not change the \ac{FT} breaking points in these campaigns, so any induced overhead was never worth it. Register Redundancy and Logic Redundancy add more hardware, but they were the only individual \acp{FTM} that changed the pass, fail, and crash results.

The All \acp{FTM} configuration gives the best \ac{FT} result in the general comparison, but its hardware cost is dominated by mechanisms that did not improve the measured \ac{FT} behaviour. The results indicate that the useful subset for these campaigns is formed by Register Redundancy and Logic Redundancy. This conclusion is specific to the tested fault model, targets, and benchmarks.

\Cref{tab:chap9-regRedundancy-status} confirms that register faults crash systems that don't have a Register Redundancy configuration able to form a correctable quorum. \(N=1\), \(M=1\) has no redundancy. \(N=2\), \(M=2\) duplicates the register, but does not contain enough replicas for a majority quorum.

Correction is the main difference between the two \(N=3\), \(M=2\) sweeps. With Correction enabled, the system passes every tested injection rate. With Correction disabled, it only passes at 1 Hz. Without Correction, a corrupted replica remains corrupted until the register is overwritten by normal execution. If later injections corrupt enough replicas before that overwrite, the voter can forward corrupted data. Correction prevents this accumulation by writing the voted value back into the disagreeing replica.

Increasing \(N\) improves the Correction-off case, because it increases the amount of corrupted replicas that the system withstands, but does not replace Correction. At 10 Hz, \(N=7\), \(M=4\) without Correction passes, while \(N=5\), \(M=3\) fails. With Correction enabled, all valid majority configurations pass at 10 Hz, starting at \(N=3\), \(M=2\), which supports enabling Correction before increasing \(N\).

With Correction enabled, each detected error is corrected in the next cycle, so the affected replica can later produce another detection. With Correction disabled, a corrupted replica remains corrupted until normal execution overwrites it. Until that happens, the same stored corruption does not keep producing new counted detections, and it can contribute to later failures. This explains the lower 10 Hz Observability values without Correction, such as 9.4813\% for \(N=5\), \(M=3\), and 17.4540\% for \(N=7\), \(M=4\). With Correction enabled, Observability stays close to 26.5\% for the \(N=3\), \(M=2\) rate sweep, which is still consistent with the aggregation limitation identified in \Cref{chap9:general-ftm-testing}.
