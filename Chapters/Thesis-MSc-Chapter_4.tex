\section{VeeRwolf EL2 Overview}
\label{chap4:el2_overview}

Imagination Technologies' Veerwolf is a reference \ac{SoC} platform in teaching and Academia \cite{rvteaching1 , rvteaching2}. Through the VeeR EL2 configuration, shown in \Cref{fig:el2}, the platform combines the EL2 core with the peripherals, interconnect, memory blocks, and build support required to execute software in simulation and on selected \acp{FPGA}.

    \caption{VeeRwolf EL2 Overview}
    \label{fig:el2}
\end{figure}

VeeR EL2 \cite{veer_el2}, surveyed in \Cref{chap2:analysis}, is a 32-bit embedded \ac{RISC-V} core that served as the first Fatori-V backbone candidate. It is delivered as a core complex with optional \ac{ICCM} and \ac{DCCM} memories, an optional instruction cache, three master ports for instruction fetch, load-store access, and debug, and one slave \ac{DMA} port for external access to core memories. The external bus protocol can be configured as 64-bit \ac{AXI} or \ac{AHB}-Lite.

\begin{figure}[h]
\centering
    \includegraphics[width=0.85\textwidth]{./Images/EL2_Full_Pipeline.png}
    \caption{VeeR EL2's Full Pipeline}
    \label{fig:el2_pipeline}
\end{figure}

The EL2 pipeline, shown in \Cref{fig:el2_pipeline}, is organized into Fetch, Decode, Execute/Memory, and Retire stages. Fetch computes the next \ac{PC} and accesses the instruction source, either the instruction cache or \ac{ICCM}. Decode aligns and decodes instructions and generates control signals to select operands. Execute/Memory performs arithmetic and control-flow operations and generates memory addresses. Retire completes memory and register writes.

The EL2 implementation separates the arithmetic and control-flow path from the memory path, shown respectively as the I-pipe and LSU-pipe in the figure. In the chosen configuration, \ac{DCCM} accesses include \ac{ECC} protection, stores use a write buffer, and bypass paths forward results from later stages back to Decode to reduce stalls.

VeeRwolf adds the \ac{SoC} blocks needed to execute software around the EL2 core complex. These include a boot ROM, system controller, \ac{UART}, \ac{SPI}, memory and peripheral interconnects, and an \ac{AXI}-to-Wishbone bridge. Since the available board at this stage was the Basys3 \cite{basys3}, the implementation followed the RVfpgaEL2-Basys3 flow, which uses the VeeRwolfX variant. This variant adds \ac{GPIO}, a \ac{PTC} timer, a controller for the four-digit seven-segment displays, \ac{AXI} RAM, \ac{JTAG} boundary-scan logic, and a clock generator.

The study on the VeeRwolf EL2 configuration was mainly supported by the RVfpgaEL2-Basys3 course, which provided examples for simulation and \ac{FPGA} execution, as well as a debug toolchain.

\section{VeeRwolf EL2 and Fault Tolerance}
\label{chap4:el2_design}

\begin{figure}[h]
\centering
    \includegraphics[width=0.9\textwidth]{./Images/EL2\_Full\_Pipe\_Faul\_Tolerance\_Baseline.png}
    \caption{Planned Implementations for VeeR EL2}
    \label{fig:el2_plan_impl}
\end{figure}

After studying the EL2 pipeline, three \ac{FTM} integration paths were planned and followed through, as shown in \Cref{fig:el2_plan_impl}. The first was register-level redundancy through \ac{TMR}, represented by the yellow registers, where each stored value would be replicated and voted before being used by the rest of the pipeline. The second was \ac{ECC} extension for storage structures, represented by the red blocks. Since \ac{ECC} already existed in the \ac{DCCM} path, the intended extension points were the register file and the \ac{ICCM}. The third was logic-level \ac{TMR} on the main address and control-flow adders, represented by the green blocks, targeting the \ac{PC} adder, the data-memory address adder, and the \ac{ALU} adder. 

When implementation began, the guided material and labs proved less useful for such low-level \ac{RTL} modification than anticipated. The planned mechanisms were harder to isolate than the diagram suggested, because EL2 contains long cross-module signal paths and tightly coupled control logic, so each modification required checking that pipeline behaviour, bypassing, memory access, and control dependencies were not being broken. To reduce this complexity before adding protection mechanisms, the design was first pruned into a simpler baseline. The branch predictor was removed, the multiplication and division paths were disabled, the bit-manipulation portion of the \ac{ALU} was removed, and both instruction and data caches were disabled so that instruction and data accesses relied on \ac{ICCM} and \ac{DCCM}. After this reduced baseline was operational, the planned implementations were attempted.

Pipeline-register \ac{TMR} was implemented by modifying the existing register wrapper to instantiate three register copies, vote their outputs with a newly developed voter, and propagate only the quorum value. \ac{TMR} was then applied to pipeline adders. The \ac{PC} adder and data-address adder were successfully triplicated because their interfaces were small and their signal boundaries were clear. The full \ac{ALU} \ac{TMR} was not completed because its integration reintroduced the same cross-module dependency problem and left unresolved implementation warnings. The final attempted extension was \ac{ECC} for additional memories beyond the already protected \ac{DCCM}. This was not completed. In EL2, close memory accesses still interact with the \ac{DMA} path, so extending memory protection required a deeper modification of the memory subsystem than the available time allowed. For this reason, the EL2-based implementation path was abandoned. The causes and consequences of this decision are discussed in \Cref{chap4:summary}.

\section{Summary}
\label{chap4:summary}

This chapter documented the first Fatori-V implementation attempt, based on the VeeRwolf EL2 platform. The EL2 core and VeeRwolfX \ac{SoC} were studied, a pruned baseline was established, and partial implementations of pipeline-register \ac{TMR}, pipeline-adder \ac{TMR}, and \ac{ECC} extension for additional memories were attempted. The tight coupling between EL2's control logic, memory interfaces, and bypass paths made each \ac{FTM} insertion significantly harder to verify than anticipated, and the available course material was insufficient to support the required depth of \ac{RTL} modification. As the scope of \ac{Fatori-V} became clearer, it also became evident that the platform required controlled \ac{FI}, metric collection, automated benchmark execution, configuration file generation, \ac{IP} generation, and Vivado-flow integration — requirements that could not be met by continuing to adapt the EL2 baseline. The EL2-based approach was therefore abandoned in favour of lowRISC's Ibex core, described in \Cref{chap5:ibex}.
