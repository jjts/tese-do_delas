\section{Fatori-V Operation Sequence}
\label{chap7:software_overview}

\begin{figure}[H]
\centering
    \includegraphics[width=0.9\textwidth]{Images/chap3_layers.jpg}
    \caption{Fatori-V Complete Overview}
    \label{fig:fatori_overview}
\end{figure}


The Software Layer is the upper layer of \ac{Fatori-V} shown in \cref{fig:fatori_overview}. It reads the user-provided run files in \texttt{runs/*.yaml} (\cref{chap3:ui_yaml}), validates and normalizes them, generates and allocates the interface files that configure the Hardware Layer (SystemVerilog headers and Vivado Tcl hooks, \cref{chap3:hwvssw}), launches the build to produce the bitstream and reports, executes the selected benchmark sessions, and packages logs, generated tables, and metrics into \texttt{results/}. It is split into the orchestrator, which drives the run and session workflow, and the \ac{FI} console (\cref{chap8:fatori_fi}). When \ac{FI} is enabled, the orchestrator generates the \ac{FI} console \ac{CLI} command per session, runs it as a subprocess, and synchronizes it with benchmark execution using the handshake described in \cref{chap7:handshake}.

The top script file is \texttt{fatori-v.py}. It iterates through all run files (see \cref{chap3:ui_yaml}) found in \texttt{runs/} (or a user-selected subset) and applies the same phased algorithm to each one, which allows users to prepare multiple experiments up front and run them sequentially with one invocation.

\ac{Fatori-V} executes each run through a fixed staged workflow. Each stage transforms the Run \texttt{.yaml} filw \cref{chap3:ui_yaml} into concrete configuration files, build inputs, benchmark sessions, and final result tables. \Cref{tab:fatori_workflow_stages} summarizes the role and output of each stage.

The Validation Stage is implemented through checks defined in \texttt{config/validation\_checks.py}, and provided by the user. One example is displayed in \cref{lst:check}. Each check inspects selected \texttt{.yaml} fields and either applies a deterministic correction or aborts the run. In strict mode, warnings without auto-corrections also abort the run. After all checks are applied, \ac{Fatori-V} writes \texttt{verified\_yaml.yaml}, which replaces the user-provided run file for the remaining stages and is stored in the final results folder.

\begin{lstlisting}[language=matlabfloz,caption={Example check in validation\_checks.py},label=lst:check]
  {'type': 'warning',
  'logic': lambda cfg: cfg['...']['icache_ecc'] and not cfg['...']['icache'],
  'message': 'ICache ECC enabled but ICache is disabled. Disabling ICache ECC.',
  'correction': lambda cfg: cfg['...'].update({'icache_ecc': 'off'})}
\end{lstlisting}

The Generation Stage creates the interface files consumed by the Hardware Layer. These include SystemVerilog headers such as \texttt{fatori\_features.svh}, \texttt{fatori\_ftm.svh}, \texttt{fatori\_logic\_mon.svh}, \texttt{fatori\_reg\_mon.svh}, \texttt{fatori\_selftest.svh}, and \texttt{fatori\_pblocks.svh}, firmware headers that allow the firmware wrapper to run only the enabled benchmarks and the Vivado \texttt{.tcl} scripts that configure the Vivado fllow. The generated files are first written under \texttt{tmp/} and are then copied into the lower-layer repositories during File Movement.

Generated files are copied according to \texttt{gen\_locations.yaml}, while user-provided hardware and software inputs are copied according to the corresponding \texttt{locations.yaml} files. Before any destination file is overwritten, the original file is backed up so the architecture tree can be restored after the run. Once the files are staged, the Build Stage runs the required Makefile targets and, when \ac{FI} is enabled, prepares the \ac{FI} console command used during execution.

The Execution Stage runs each enabled benchmark as a separate session. A session is considered valid only if the process exits cleanly, the benchmark succeeds, and a valid \texttt{metrics.txt} file is produced. When \ac{FI} is enabled, the session controller also launches the \ac{FI} console as a parallel subprocess and relies on the synchronization mechanism described in \cref{chap7:handshake}.

The Results Stage converts raw measurements into result tables. Benchmark data is extracted from \texttt{metrics.txt}, while implementation data is parsed from Vivado reports. Direct metrics are copied or differenced from raw values, and indirect metrics are computed from formulas defined by the user in \texttt{config/metrics\_definitions.py}. The full list of direct and indirect metrics is provided in \cref{chap9:computed_metrics}. A metric definition example is displayed in \cref{lst:metrics}.

\section{Pblock Sizing Model}
\label{chap7:pblock}

To inject faults into a specific hardware module, the \ac{FI} tool must know where that module is placed in the \ac{FPGA}. \ac{Fatori-V} solves this by constraining each injectable target to a pblock and storing its bounds in the System Dictionary used by the \ac{FI} tool. These bounds are later used to select configuration memory addresses that belong to the target region, as described in \cref{chap8:fi_acme_impl}.

The size of each hardware module depends on the active Ibex configuration, including \ac{ISA} extensions, multiplier type, bit manipulation variant, cache support, branch logic, writeback stage, \ac{FI} instrumentation, and enabled \acp{FTM}. A static table would not cover all possible combinations, especially for parameters such as the M-of-N replication factor, which can be virtually any integer number. The Pblock Sizing Model estimates the required area of each target from the configuration used in each run.

The model is used for the following pblock targets: \ac{ALU}, \ac{LSU}, Controller, Decoder, IF Stage, ID Stage, Multiplier, Prefetch Buffer, WB Stage, Branch Predictor, and Fault Manager.







% -----------------------------------------------------------------------------
\subsection{Sizing Model}
\label{chap7:pblock_sizing_model}

The model estimates the \ac{LUT} and \ac{FF} usage of each pblock target. Each target starts from a measured baseline and adds the overhead of the active hardware features. The baseline with architectural features is defined in \cref{eq:pblock_target_base}.

\begin{equation}
A_t =
B_t
+
\sum \Delta^{\text{feat}}_t
\label{eq:pblock_target_base}
\end{equation}

In \cref{eq:pblock_target_base}, \(B_t\) is the measured \ac{LUT} baseline of target \(t\), and \(\sum \Delta^{\text{feat}}_t\) is the sum of the enabled architectural feature overheads. Negative feature effects are set to zero, preventing the model from estimating  pblock sizes below the measured baseline.

Logic M-of-N replicates the complete target. This includes the combinational logic and the registers inside the target. Register M-of-N overhead, if the \ac{FTM} is enabled, is included within each logic replica. The \ac{LUT} estimate is computed with \cref{eq:pblock_lut_model}.

\begin{equation}
\text{LUT}_t =
N_{\text{logic}}
\left(
A_t
+
\Delta^{\text{RegMon}}_t(N_{\text{reg}})
\right)
+
V_t(N_{\text{logic}})
\label{eq:pblock_lut_model}
\end{equation}

In \cref{eq:pblock_lut_model}, \(N_{\text{logic}}=1\) when Logic M-of-N is disabled. \(V_t(N_{\text{logic}})\) is the voter cost of the Logic M-of-N wrapper and is set to zero when Logic M-of-N is disabled.

Register M-of-N mainly increases the number of \acp{FF}, because it replicates stored state. However, the generated wrappers, voters, correction logic, and control logic can also use \acp{LUT}. For this reason, the model includes a Register M-of-N \ac{LUT} term when measured data showed a relevant \ac{LUT} cost. This term is computed with \cref{eq:pblock_reg_mon}.

\begin{equation}
\Delta^{\text{RegMon}}_t(N_{\text{reg}})
=
P_t(N_{\text{reg}})
-
A_t
\label{eq:pblock_reg_mon}
\end{equation}

In \cref{eq:pblock_reg_mon}, \(P_t(N_{\text{reg}})\) is the measured or fitted total \ac{LUT} count of target \(t\) with Register M-of-N enabled. When Register M-of-N is disabled, or when the measured \ac{LUT} overhead was not relevant, \(\Delta^{\text{RegMon}}_t(N_{\text{reg}})=0\).

The \ac{FF} model follows the same structure. The model starts from the measured \ac{FF} baseline, adds the \ac{FF} effects of the enabled architectural features, and applies the \ac{FF} cost of Register M-of-N when enabled. Since XCKU040 slices contain 8 \acp{LUT} and 16 \acp{FF}, the final pblock size is set by the most restrictive resource, as shown in \cref{eq:pblock_slice_binding}.

\begin{equation}
\text{Slices}_{t}
=
\left\lceil
M_t
\cdot
\max
\left(
\left\lceil \frac{\text{LUT}_{t}}{8} \right\rceil,
\left\lceil \frac{\text{FF}_{t}}{16} \right\rceil
\right)
\right\rceil
\label{eq:pblock_slice_binding}
\end{equation}

\(M_t\) is the safety margin used to account for placement and routing constraints.

\subsection{Parameter Extraction}
\label{chap7:pblock_study}

\section{Benchmark Porting}
\label{chap7:benchmarks}
\ac{Fatori-V} natively integrates four benchmark programs than can be executed sequentially in the same run: IObundle's Hello World, CoreMark \cite{coremark}, Embench-IoT \cite{embench-iot}, and Fatori Stress (designed in this work to stress the exact Ibex targets that \ac{Fatori-V} can inject into). All benchmarks follow the same interface and folder format under \texttt{benchmarks/}: a folder whose name matches the benchmark key in the run \texttt{.yaml}, a \texttt{benchmark.h} that exposes the benchmark entry point as \texttt{BENCHMARK\_MAIN} (called by the firmware wrapper, see \cref{chap5:iob-soc-ibex}), the \texttt{repo/} directory containing a clone of the official benchmark repository (when applicable), a \texttt{Makefile} with a \texttt{prepare/setup} target that copies the required upstream sources into \texttt{code/} for compilation, an optional \texttt{requirements.yaml} that declares extra build/link flags, and the benchmark-specific wrapper/port files required to run in a baremetal environment (e.g., local \texttt{printf} implementation instead of OS/stdio).  


% -------------------------------------------------------------
\subsection{CoreMark Benchmark}
\label{chap7:coremark}
CoreMark \cite{coremark} is an industry-standard \ac{CPU} benchmark commonly reported in the state of the art, so including it gives \ac{Fatori-V} a reference workload that is easy to compare against other \ac{RISC-V} results. 

In \ac{Fatori-V}, CoreMark is configured by a single \texttt{.yaml} iterations parameter  compiled into \texttt{bench\_config.h} as \texttt{ITERATIONS}. The Results Phase exports the standard CoreMark report fields, such as  \texttt{coremark\_size}, \texttt{total\_ticks}, \texttt{total\_time\_secs}, \texttt{iterations\_per\_sec}, \texttt{iterations}, \texttt{memory\_location}, and the \ac{CRC} fields \texttt{seedcrc}, \texttt{crclist}, \texttt{crcmatrix}, \texttt{crcstate}, and \texttt{crcfinal}. These fields are used to calculate the CoreMark Score and are merged with the remaining measured metrics to generate the results tables.

CoreMark runs four kernels: linked-list processing, matrix operations, a small state machine, and \ac{CRC} computation. The official porting point for new platforms is \texttt{core\_portme.c/h}, which must define the \texttt{ee\_*} datatypes, configuration constants such as \texttt{MEM\_METHOD}, \texttt{SEED\_METHOD}, \texttt{HAS\_STDIO}, \texttt{HAS\_PRINTF}, and \texttt{MULTITHREAD}, and the timing interface. In the \ac{Fatori-V} integration, \texttt{core\_portme.h} configures the software by choosing static memory with \texttt{MEM\_STATIC} and disabling floats with \texttt{HAS\_FLOAT}. The \texttt{core\_portme.c} file sizes the native volatile seeds to select a \texttt{PERFORMANCE\_RUN} and redirects \texttt{ee\_printf} to a locally implemented version. The developed \texttt{coremark\_main.c} file resolves the dual-main issue by renaming CoreMark's main through a preprocessor alias before including the upstream \texttt{core\_main.c}. The locally developed \texttt{coremark\_metrics.c/h} files implement \texttt{coremark\_printf} as a dual-sink logger, parse the captured CoreMark report lines, and export the extracted fields as \texttt{"[benchmark] key,value"} lines appended to \texttt{metrics.txt}.


% -------------------------------------------------------------
\subsection{Embench-IoT Benchmark}
\label{chap7:embench-iot}
Embench-IoT \cite{embench-iot} is a modern embedded benchmark suite designed for baremetal use and for producing a single consistent score from a diverse set of small programs.

In \ac{Fatori-V}, Embench is configured through the run \texttt{.yaml} file by selecting which sub-benchmarks are enabled. This selection is compiled into \texttt{bench\_config.h} as one 0/1 macro per sub-benchmark. The Results Phase exports \texttt{total\_benchmarks}, \texttt{total\_passed}, \texttt{embench\_score}, and, for each executed sub-benchmark, its cycles, pass flag, and ratio. The current IOb-SoC-Ibex software memory budget cannot link the full suite simultaneously, so a validation check (see \cref{chap7:validation_stage}) enforces a limit of at most 10 enabled sub-benchmarks and automatically disables the excess to keep the build runnable.

Each Embench-IoT program follows an initialize, benchmark, and verify structure. The suite score is computed by comparing each program's measured cycle count against the baseline cycle counts distributed with Embench. To implement it in \ac{Fatori-V}, the \texttt{embench\_benchmarks.c} wrapper pulls the selected Embench programs into the build and renames their entry functions so multiple sub-benchmarks can be linked together without symbol collisions. The \texttt{embench\_main.c} wrapper runs each enabled sub-benchmark through its renamed \texttt{initialise\_benchmark\_*} and \texttt{benchmark\_*} functions, measures cycles with \texttt{mcycle}/\texttt{mcycleh} reads, verifies results with \texttt{verify\_benchmark\_*}, computes ratios against \texttt{baseline\_cycles} from \texttt{embench\_data.h}, and exports all values to \texttt{metrics.txt}. The \texttt{boardsupport.c/h} files provide the minimal hooks expected by the upstream suite, and \texttt{requirements.yaml} declares the required \texttt{-lm} link flag and compile ordering so math-heavy programs can run under soft-float.

\subsection{Fatori Fault Tolerance Benchmark}
\label{chap7:fatori_stress}
Fatori Stress was developed specifically for this thesis to create deterministic, self-validating workloads that can be aligned with \ac{FI} campaigns (see \cref{chap8:fatori_fi}) while stressing specific Ibex targets. The benchmark executes the same operations twice: a forward run from iteration $0\rightarrow (N-1)$ followed by a backward run from $(N-1)\rightarrow 0$. At each iteration, it computes a per-iteration value and XORs it into a running checksum. XOR is self-inverse, which means that a fault-free execution returns exactly to the initial checksum after the backward pass, while any corruption breaks reversibility and flags the run as FAIL. Operands are derived only from the iteration index, so the workload changes with the requested iteration count but remains fully reproducible.

The user-requested configuration is compiled through the generated \texttt{bench\_config.h}: \texttt{FATORI\_ITERATIONS} selects $N$, and per-target macros select which module workloads are executed. This version implements seven targets: the \ac{ALU}, Multiplier/Divider, Decoder, Controller, \ac{LSU}, Branch Predictor, and the Instruction Decompressor. \ac{Fatori-V}'s Results Phase (see \cref{chap7:results_stage}) records benchmark-specific fields: enabled target list, iteration count, initial, forward, and backward checksum values, validation status (\texttt{PASS}, \texttt{FAIL}, or \texttt{SKIPPED}), and the first iteration where a mismatch is detected.

Each Fatori Stress iteration calls a dispatcher that invokes one stress function per enabled target and XOR-accumulates their return values. \texttt{alu\_stress} mixes independent arithmetic and logic operations with dependent chains, variable shifts, and signed/unsigned comparisons. \texttt{multiplier\_stress} uses \texttt{mul}/\texttt{div} operations when the toolchain \ac{ISA} enables RV32M. Otherwise, it executes a shift-and-add multiply and a long-division loop. \texttt{decoder\_stress} alternates instruction shapes by combining register-register operations, immediate operations, and volatile array accesses. \texttt{controller\_stress} stresses control flow with nested conditionals, backward-branch loops, and noinline helper calls. \texttt{lsu\_stress} writes a known word pattern into a static buffer and reads it back as byte, halfword, and word accesses. \texttt{branch\_predict\_stress} generates predictable branch patterns. \texttt{compressed\_stress} is used when the firmware is compiled with RV32C enabled, forcing the fetch stream to contain mixed 16-bit and 32-bit encodings.

\section{Synchronization, Logging and Metrics Computing}
\label{chap7:sw_logging}

\ac{Fatori-V} uses a file-based handshake to align the firmware’s “start of benchmark” with the \ac{FI} console’s “start of campaign”. The firmware wrapper creates a small sync file (using the provided \ac{UART} file-transfer functions) containing the string "WAIT", the orchestrator passes the path of that file to the \ac{FI} console command line, and the \ac{FI} console freezes until the file appears and then overwrites its content with "READY". Firmware execution is held in a polling loop that repeatedly reads the file and only proceeds once "WAIT" becomes "READY", at which point the benchmark body starts.

To mitigate \ac{UART} induced delays, the firmware wrapper executes all setup work (\ac{UART} init, metric pre-sampling, and any preliminary reads) before entering the "WAIT"/"READY" protocol loop. 

From the \ac{FI} perspective, the campaign begins as soon as "READY" is written, because all \ac{FI} setup (TargetPool building, ACME algorithms, and preflight tests) is completed before the handshake protocol. Since the firmware only observes that update on the next polling iteration, there can be a small propagation delay between “\ac{FI} starts” and “benchmark begins”, so the first few injections may occur before the benchmark actually begins executing. In practice, campaigns should use sufficiently large injection pools (typically $>500$ injections) so that this initial misalignment has less impact on the overall statistics.


Logging is implemented as a centralized event system under \texttt{scripts/logging/} and configured by two config files (see \cref{chap3:ui_config}): \texttt{log\_levels.py} selects which events are printed for each verbosity level (minimal/normal/verbose), and \texttt{messages\_formats.py} defines the displayed messages. Scripts do directly print strings, they emit an event ID plus a small set of fields, and the logger decides whether to print the event and how to format it. This keeps timestamps, prefixes, and sinks consistent across the project and implements increased system customization. Each event is associated with a verbosity level, so users can precisely choose which events are shown on the terminal and written to the log for minimal, normal, and verbose runs. An example is provided in \cref{lst:log}.

\begin{lstlisting}[language=matlabfloz,caption={Example log event implementation},label=lst:log]
from scripts.logging.logger import log_event
log_event('DEBUG', 'FILE\_ALLOCATION', num\_alloc\_files")
\end{lstlisting}


For each run, \ac{Fatori-V} extracts \texttt{metrics.txt}, produced by the firmware, which contains the RAW measured data: (i) snapshots of Ibex native hardware-performance-monitoring counters (see \cref{chap5:ibex_overview}) and the added \ac{Fatori-V} \acp{CSR} (see \cref{chap6:metrics_hw}), sampled both before and after each benchmark, and (ii) benchmark-specific values reported by the benchmark itself (through per-benchmark parsers/wrappers implemented for \ac{Fatori-V}, see \cref{chap7:benchmarks}).

In the Results Stage, higher-level metrics are computed from this raw data using the user-provided metric definitions in \texttt{config/metrics\_definitions.py} (see \cref{chap3:ui_results}), which include a set of default definitions and can be extended with additional formulas. \ac{Fatori-V} builds the results tables, by appending all raw and computed metrics, along with the extracted data from Vivado reports. \Cref{tab:metrics_all} lists all metrics exported by \ac{Fatori-V} in all experiments.
