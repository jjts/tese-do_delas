\section{Fatori-V Operation Sequence}
\label{chap7:software_overview}

A \ac{Fatori-V} run starts from a user-defined \texttt{.yaml} file placed under \texttt{runs/}. The run file defines the hardware setup, benchmark selection, build options, and \ac{FI} parameters. From that file, \ac{Fatori-V} prepares the configured experiment by generating the required files, staging them in the Py2HWSW Setup Directory and benchmark folders, launching the build and execution flow, and collecting the data used in the final result tables. \Cref{fig:fatori_overview} summarizes the run flow.

\begin{figure}[h]
\centering
    \includegraphics[width=0.9\textwidth]{Images/chap3_layers.png}
    \caption{\ac{Fatori-V} run flow from input run files to result tables.}
    \label{fig:fatori_overview}
\end{figure}

The entry script, \texttt{fatori-v.py}, reads the selected run files and applies the same sequence to each one. When \ac{FI} is enabled, the same flow also prepares the \ac{FI} command and launches the \ac{FI} tool during benchmark execution.

The Validation Stage checks the input \texttt{.yaml} file before any generated file is written. The checks are defined in \texttt{config/validation\_checks.py}, which can be extended. Each check inspects selected fields and either applies a deterministic correction or aborts the run. After validation, \ac{Fatori-V} rewrites the input \texttt{.yaml} file into \texttt{verified\_yaml.yaml}, which is used by the remaining stages and stored in the final results folder.

The Generation Stage translates the verified configuration into the interface macro files required by the Py2HWSW Layer described in \cref{chap6:fatori_hardware}. It also generates firmware headers for benchmark selection and Vivado \texttt{.tcl} scripts for the Vivado build flow.

The File Movement Stage places generated and user-provided files in the directories expected by the Py2HWSW build. Generated files are copied according to \texttt{gen\_locations.yaml}. User-provided hardware and software files are copied according to the corresponding \texttt{locations.yaml} files. Before overwriting a destination file, \ac{Fatori-V} creates a backup so the repository can be restored after the run.

The Build Stage runs the IOb-\ac{SoC}-Ibex Makefile targets from the Py2HWSW Setup Directory. It builds the \ac{FPGA} image, prepares the selected benchmarks, and stores the implementation reports later used for hardware metrics. When \ac{FI} is enabled, this stage also prepares the command used to launch the \ac{FI} tool during execution.

The Execution Stage runs each enabled benchmark as a separate session. During each session, the firmware samples the hardware metrics before and after benchmark execution and sends them to the host, where they are written to \texttt{metrics.txt}. When \ac{FI} is enabled, the session controller launches the \ac{FI} tool as a parallel process and synchronizes it with the benchmark.

The Results Stage collects benchmark outputs, \texttt{metrics.txt} files, \ac{FI} logs, and implementation reports from the run directory. These files are stored under \texttt{results/} and used to generate the final result tables.

\section{Benchmark Porting}
\label{chap7:benchmarks}

\ac{Fatori-V} integrates four benchmark programs that can run sequentially in the same experiment. These are IObundle's Hello World, CoreMark, Embench-IoT, and Fatori Stress. Hello World is used as a simple execution check. CoreMark and Embench-IoT provide standard performance benchmarks. Fatori Stress was developed in this work as a checksum-based way to evaluate \ac{FT} on Ibex.

All benchmarks were ported following the same folder structure under \texttt{benchmarks/}. Each benchmark has one folder whose name matches the key used in the run \texttt{.yaml}. The folder requires a \texttt{benchmark.h} file that defines \texttt{BENCHMARK\_MAIN} as the benchmark's main function, a \texttt{repo/} folder with the upstream source when applicable, a \texttt{Makefile} that generates the sources from \texttt{repo/}, and wrapper files that adapt the benchmark to the bare-metal firmware environment. Some benchmarks also include \texttt{requirements.yaml} to declare extra compile or link flags.

Benchmark options exposed by each port are routed to the run \texttt{.yaml}, so the user can configure benchmark parameters together with the remaining experiment settings.




% -------------------------------------------------------------
\subsection{CoreMark Benchmark}
\label{chap7:coremark}

CoreMark is included as a reference benchmark for comparing \ac{CPU} implementations. In \ac{Fatori-V}, the number of iterations is selected in the run \texttt{.yaml} file and compiled into \texttt{bench\_config.h} as \texttt{ITERATIONS}.

CoreMark evaluates linked-list handling, matrix operations, state-machine execution, and \ac{CRC} computation. The benchmark reports execution time, iteration count, iterations per second, memory configuration, and \ac{CRC} values. \ac{Fatori-V} parses these fields from the CoreMark report and stores them in \texttt{metrics.txt}. The CoreMark score is computed as the number of completed iterations divided by the measured execution time.

The port uses the official \texttt{core\_portme.c/h} files. The header selects static memory allocation and disables floating-point support. The source file redirects \texttt{ee\_printf} to the local print implementation.

The wrapper \texttt{coremark\_main.c} adapts the upstream entry point to the common benchmark interface. It renames CoreMark's original main function before including the upstream \texttt{core\_main.c}, allowing the firmware wrapper to call it through \texttt{BENCHMARK\_MAIN}. The files \texttt{coremark\_metrics.c/h} capture the CoreMark report, extract the relevant fields, and write them to \texttt{metrics.txt}.






% -------------------------------------------------------------
\subsection{Embench-IoT Benchmark}
\label{chap7:embench-iot}

Embench-IoT executes a set of embedded sub-benchmarks sequentially and combines their results into a score commonly used to evaluate \acp{CPU}. In \ac{Fatori-V}, the user selects the enabled sub-benchmarks through the run \texttt{.yaml} file. This selection is compiled into \texttt{bench\_config.h}, with one enable macro per sub-benchmark.

Each selected Embench-IoT sub-benchmark prepares its input data, runs the timed benchmark code, and checks whether the produced output is correct. \ac{Fatori-V} records the executed sub-benchmarks, pass count, Embench-IoT score, cycle count, pass flag, and ratio between the measured cycle count and the reference cycle count stored in \texttt{embench\_data.h}. Because the software memory available on the board used in this work is limited, \ac{Fatori-V}'s validation restricts each run to ten enabled sub-benchmarks. If the run file enables more than ten, the excess entries are disabled before the build. This limit is defined in \texttt{validation\_checks.py}.

The file \texttt{embench\_benchmarks.c} includes the selected Embench-IoT programs and renames their \texttt{main} functions to avoid symbol conflicts. The file \texttt{embench\_main.c} runs each enabled sub-benchmark, measures cycles using the hardware cycle counters, verifies the result, computes the ratio, and writes the measured values to \texttt{metrics.txt}. The files \texttt{boardsupport.c/h} provide the platform hooks required by Embench-IoT, and \texttt{requirements.yaml} adds the \texttt{-lm} link flag required by math-based sub-benchmarks.

\subsection{Fatori Native Benchmark}
\label{chap7:fatori_stress}

Fatori Stress was developed in this work to provide a deterministic workload for \ac{FI} campaigns. It is self-validating and can be configured to exercise specific Ibex targets.

The benchmark executes the selected workload twice. The first pass runs from iteration \(0\) to \(N-1\). The second pass runs from iteration \(N-1\) back to \(0\). Each iteration computes a value from the iteration index and XORs it into a running checksum. Since XOR is self-inverse, a fault-free execution returns to the initial checksum after the backward pass. If a fault changes the computed values, the final checksum does not match and the benchmark reports \texttt{FAIL}.

The generated \texttt{bench\_config.h} file configures the benchmark. The macro \texttt{FATORI\_ITERATIONS} sets \(N\), and one macro per target selects which stress functions are executed. The current implementation supports the \ac{ALU}, Multiplier/Divider, Decoder, Controller, \ac{LSU}, Branch Predictor, and Instruction Decompressor.

Each iteration calls a dispatcher that executes the enabled target workloads and combines their return values into the checksum. The \ac{ALU} workload exercises arithmetic, logic, shifts, and signed and unsigned comparisons. The multiplier workload uses multiplication and division when RV32M is enabled, and falls back to software loops otherwise. The decoder workload alternates instruction formats through register-register operations, immediate operations, and volatile memory accesses. The controller workload exercises control flow through nested conditions, loops, and non-inlined helper calls. The \ac{LSU} workload writes a known pattern to a static buffer and reads it back using byte, halfword, and word accesses. The branch predictor workload produces branch patterns. The compressed instruction workload is used when RV32C is enabled, forcing the instruction stream to include mixed 16-bit and 32-bit encodings.

During execution, Fatori Stress records the enabled target list, iteration count, initial checksum, forward checksum, backward checksum, validation status, and first mismatch iteration when a mismatch is detected.

\section{Synchronization, Logging and Metrics Computing}
\label{chap7:sw_logging}

\ac{Fatori-V} synchronizes the \ac{FI} campaign with benchmark execution using a file-based handshake. Before starting the benchmark, the firmware running on the board requests the host \ac{PC} to create a synchronization file containing \texttt{WAIT}. The \ac{FI} tool receives the path to this file, waits until it is created, and writes \texttt{READY} when it is ready to inject. The firmware polls the same file and starts the benchmark only after reading \texttt{READY}.

The firmware and the \ac{FI} tool complete their setup before the synchronization step, so the benchmark and the campaign can start immediately after the handshake. The firmware initializes the \ac{UART}, samples the pre-benchmark metrics, prepares the benchmark, and writes \texttt{WAIT}.

A small timing offset can still exist. The \ac{FI} tool starts the campaign after writing \texttt{READY}, while the firmware only observes that update on its next polling iteration. A short delay is inserted after synchronization, but the first injections may still occur slightly before the benchmark begins executing. The experiments use large injection pools, so this offset has limited impact on the final campaign statistics.

Logging is centralized through a dedicated event system. Scripts do not print messages directly. They emit a log event with an event identifier and the fields needed to build the message. The logger then decides whether the event is printed and how it is formatted.

Logging can be tuned through two configuration files that define which events are shown for each verbosity level and the corresponding message text. This lets users control how much information \ac{Fatori-V} reports during a run, while keeping prefixes, timestamps, terminal output, and log files consistent across the platform.


\ac{Fatori-V} generates result tables using measured and computed values. During the Results Stage, it loads \texttt{config/metrics\_definitions.py}, computes every defined metric for each benchmark session, and adds the values to the output tables. Users can extend this file with new metrics. Metrics measured in hardware are called direct metrics and must be written by the firmware to \texttt{metrics.txt}. Once defined, direct metrics can be used to compute indirect metrics, if their formulas are added to \texttt{config/metrics\_definitions.py}.

Some direct metrics are sampled before and after the benchmark. For these metrics, the firmware writes both values to \texttt{metrics.txt}, and \ac{Fatori-V} stores the post-minus-pre difference in the result tables.

The Results Stage also parses Vivado implementation reports to add hardware metrics. The final tables combine direct metrics, indirect metrics, benchmark-specific fields, \ac{FI} metrics, and implementation data. \Cref{app_b:metrics} lists the metrics exported by \ac{Fatori-V} in its current version.

\section{Pblock Sizing Model}
\label{chap7:pblock}


To inject faults into a selected hardware target, the \ac{FI} tool requires the pblock coordinates that contain that target. Users can define these pblocks manually, but this would make each new configuration require manual placement work. \ac{Fatori-V} implements the Pblock Sizing Model to automate this step. The model estimates the minimum pblock size required to contain each target from the selected hardware configuration, then uses that size to generate the pblock constraints used by the \ac{FI} flow.

The required pblock size depends on the active Ibex configuration. The main factors are the selected \ac{ISA} extensions, multiplier type, bit manipulation variant, cache support, branch logic, writeback stage, \ac{FI} instrumentation, enabled \acp{FTM}, and \ac{FTM} configuration. 

A fixed size table would not cover the supported configurations, especially when any M-of-N \ac{FTM} is enabled, since it accepts any replication factor.

\ac{Fatori-V} implements the model for the following pblock targets: \ac{ALU}, \ac{LSU}, Controller, Decoder, IF Stage, ID Stage, Multiplier, Prefetch Buffer, WB Stage, Branch Predictor, and Fault Manager.

The model estimates \ac{LUT} and \ac{FF} usage independently, using the same structure for both resources. For each resource \(R \in \{\text{\ac{LUT}}, \text{\ac{FF}}\}\), the estimate starts from a measured baseline and adds the contributions of the enabled architectural features, as shown in \Cref{eq:pblock_target_base}.

\begin{equation}
A_{t,R} =
B_{t,R}
+
\sum \Delta^{\text{feat}}_{t,R}
\label{eq:pblock_target_base}
\end{equation}

In \Cref{eq:pblock_target_base}, \(B_{t,R}\) is the measured baseline of target \(t\) for resource \(R\), and \(\Delta^{\text{feat}}_{t,R}\) is the contribution of each enabled architectural feature. Negative measured contributions are set to zero, so the model does not estimate pblock sizes below the measured baseline.

Feature contributions with a fixed set of values, such as multiplier type or bit manipulation variant, are stored as table parameters. Effects that depend on a variable parameter, such as the M-of-N replication factor \(N\), are fitted from measurement traces instead of being listed for every possible value.


Despite Register M-of-N mainly increasing \ac{FF} usage, it can also add \acp{LUT} through wrappers, voters, correction logic, and control logic. For each target and resource, this contribution is modelled as proportional to the number of additional register replicas, as shown in \Cref{eq:pblock_reg_mon}.

\begin{equation}
\Delta^{\text{RegMon}}_{t,R}(N_{\text{reg}})
=
(N_{\text{reg}} - 1)
\cdot
C^{\text{RegMon}}_{t,R}
\label{eq:pblock_reg_mon}
\end{equation}

In \Cref{eq:pblock_reg_mon}, \(C^{\text{RegMon}}_{t,R}\) is the measured contribution of one additional register replica for target \(t\) and resource \(R\). This coefficient is extracted from implementation measurements. When Register M-of-N is disabled, when the target has no wrapped registers, or when the measured contribution was not relevant for resource \(R\), \(\Delta^{\text{RegMon}}_{t,R}(N_{\text{reg}})=0\).

Logic M-of-N replicates the complete target. This includes the feature-dependent logic represented by \(A_{t,R}\), and the Register M-of-N contribution when it is enabled. Since \(A_{t,R}\) already represents one target instance, the Logic M-of-N contribution adds only the extra replicas and the wrapper voter cost, as shown in \Cref{eq:pblock_logic_mon}.

\begin{equation}
\Delta^{\text{LogicMon}}_{t,R}(N_{\text{logic}},N_{\text{reg}})
=
(N_{\text{logic}}-1)
\left(
A_{t,R}
+
\Delta^{\text{RegMon}}_{t,R}(N_{\text{reg}})
\right)
+
V_{t,R}(N_{\text{logic}})
\label{eq:pblock_logic_mon}
\end{equation}

In \Cref{eq:pblock_logic_mon}, \(N_{\text{logic}}=1\) when Logic M-of-N is disabled. \(V_{t,R}(N_{\text{logic}})\) is the resource cost of the Logic M-of-N wrapper and voter for resource \(R\), set to zero when Logic M-of-N is disabled.

The final estimate for each resource is defined in \Cref{eq:pblock_resource_model}.

\begin{equation}
\hat{R}_{t,R}
=
A_{t,R}
+
\Delta^{\text{RegMon}}_{t,R}(N_{\text{reg}})
+
\Delta^{\text{LogicMon}}_{t,R}(N_{\text{logic}},N_{\text{reg}})
\label{eq:pblock_resource_model}
\end{equation}

After the \ac{LUT} and \ac{FF} estimates are computed, they are converted into the number of required Kintex Ultrascale XCKU040 \cite{xcku040} slices. Since each slice contains 8 \acp{LUT} and 16 \acp{FF}, the pblock size is set by the resource that requires more slices, as shown in \Cref{eq:pblock_slice_binding}.

\begin{equation}
\text{Slices}_{t}
=
\left\lceil
M_t
\cdot
\max
\left(
\left\lceil \frac{\hat{R}_{t,\text{\ac{LUT}}}}{8} \right\rceil,
\left\lceil \frac{\hat{R}_{t,\text{\ac{FF}}}}{16} \right\rceil
\right)
\right\rceil
\label{eq:pblock_slice_binding}
\end{equation}

In \Cref{eq:pblock_slice_binding}, \(M_t\) is a multiplicative pblock sizing margin. It is applied once after the \ac{LUT} and \ac{FF} estimates are converted to slices, reserving extra placement space for routing and implementation constraints.

The model parameters were extracted from implementation reports generated for the Kintex Ultrascale XCKU040 \cite{xcku040} \ac{FPGA}. 


\ac{Fatori-V} was used to automate the parameter extraction experiments. Since the Pblock Sizing Model was not yet implemented, these runs used manually written Vivado \texttt{.tcl} files that assigned each target to fixed pblocks. The pblocks were intentionally oversized, so implementation would not fail because of an underestimated target region. Manually written Vivado \texttt{.tcl} files were placed in the expected output location, bypassing the Pblock Sizing Model for these runs.

Each run enabled a controlled hardware configuration and used the normal \ac{Fatori-V} build flow to implement the design. Vivado hierarchical utilization reports were then used to measure the \ac{LUT} and \ac{FF} usage of each target, while pblock reports showed how much area had been reserved. These measurements were used to tune the baseline terms, feature contributions, and M-of-N terms used by the model.

The study included baseline configurations, single-feature configurations, M-of-N sweeps, and combined configurations. Baseline configurations measured each target with optional features and \acp{FTM} disabled. Single-feature configurations enabled one parameter at a time. M-of-N sweeps varied \(N\) for Register M-of-N and Logic M-of-N. Combined configurations checked whether the extracted terms still produced usable pblock sizes when features and \acp{FTM} were enabled together.

\section{Summary}
\label{chap7:summary}

This chapter presented the campaign orchestration component of the Fatori-V layer. The operation sequence was described, showing how \ac{Fatori-V} translates \texttt{.yaml} run files into configured hardware and software builds and drives the full experiment lifecycle. The Pblock Sizing Model was presented, enabling module-targeted configuration memory injections. Benchmark porting was described for CoreMark, Embench-IoT, and the Fatori Native benchmark. Finally, the synchronisation, logging, and metrics infrastructure was presented, covering the file-based handshake, the event-driven logging system, and the result-table generation pipeline. \Cref{chap8:fatori_fi} describes the fault injection tool, also part of this layer.
