\section{General Features and Overview}
\label{chap3:fatori-v_overview}

\begin{figure}[H]
\centering
    \includegraphics[width=0.7\textwidth]{Images/chap3_layers_simple.png}
    \caption{Fatori-V Overview}
    \label{fig:fatori_simple_overview}
\end{figure}


\ac{Fatori-V} is an open-source Python platform designed to support the implementation and evaluation of \acp{FTM}. It allows users to run experiments on a \ac{RISC-V} \ac{SoC} by providing a simple, compact, human-readable \ac{UI} and automating the remaining steps. \Cref{fig:fatori_simple_overview} displays its two-layer system and the main components involved.

The user sets up the platform using input and configuration files. For each
experiment, a \texttt{.yaml} file is needed. \ac{Fatori-V} generates MACRO
files, which are header files for SystemVerilog, firmware, benchmark
configuration, and auxiliary scripts such as Vivado TCL hooks.

The Py2HWSW layer compiles the design, produces the bitstream, and programs
the \ac{FPGA}. The platform executes benchmark sessions on the board and, when
enabled, synchronizes them with an \ac{FI} campaign, collecting logs and
creating results tables. One run represents one architecture configuration,
which contains one session for each benchmark software executing in that
architecture. Some other features of \ac{Fatori-V} are described below.
Fatori-V is an open-source Python platform engineered to facilitate the
implementation and evaluation of Fault Tolerance Mechanisms (FTMs) within RISC-V
System-on-Chip (SoC) architectures. By providing a compact, human-readable
interface that automates complex workflows, the platform streamlines the
execution of experiments on FPGA hardware. As illustrated in Figure 3.1, the
system architecture operates across two primary layers, managing the transition
from user-defined configurations to automated hardware deployment and data
collection.

The operational workflow begins with the user providing input and configuration
files, specifically through .yaml files. Fatori-V interprets these files to
generate the necessary MACRO headers for SystemVerilog, firmware, benchmark
configurations, and auxiliary Vivado TCL scripts. Subsequently, the Py2HWSW
layer automates the compilation, bitstream generation, and FPGA programming
processes. During execution, the platform orchestrates benchmark sessions and
integrates optional Fault Injection (FI) campaigns, capturing logs and
synthesizing results into structured tables. Within this framework, a single
"run" represents a specific architecture configuration, encompassing multiple
benchmark sessions to ensure comprehensive testing.

The platform offers extensive configurability across four primary
domains. First, Architectural Configuration allows users to define performance
features such as Instruction Cache settings and RISC-V ISA
extensions. Additionally, it provides control over monitor depth and enables
dynamic sizing of PBlocks for native Ibex modules, allowing for precise control
over logic utilization.

Second, the Fault Tolerance and Injection capabilities support the selection and
parameterization of both native Ibex and platform-specific FTMs, including
M-of-N redundancy schemes. Complementing this, the Fault Injection module allows
for temporal and spatial distribution of errors—ranging from uniform to burst
distributions across various hardware targets—enabling unified injection into
both configuration memory and registers.

Third, the Batch Execution and Automation features maximize efficiency and
reproducibility. The platform supports sequential batch processing of
experiments and multiple benchmark sessions per run. Features such as fixed-seed
trees, input auto-fixes, and "dry mode" debugging ensure that experiments remain
reliable and traceable. Users can also incorporate standard benchmarks, such as
CoreMark and Embench-IoT, or provide file overrides to bypass specific build
steps.

Finally, the Data Logging and Reporting functionality automates the storage of per-run and per-session logs, governed by configurable verbosity levels and centralized message control. It facilitates performance analysis by exporting results to .csv and .xlsx formats, mirroring build artifacts for future inspection, and calculating derived metrics through centralized configuration files.

\section{Implementation Layers and Platform Structure}
\label{chap3:hwvssw}

As shown in \cref{fig:fatori_simple_overview}, the \ac{Fatori-V} Layer communicates with the Py2HWSW layer through MACROfiles. Among these files, an important one is the \texttt{fatori\_features.svh} file, shown below.

\begin{lstlisting}[language=matlabfloz,caption={Example of build-time feature macros generated by the software layer.},label=lst:fatori_features]
`__FATORI_MACRO_DEF(FATORI_ICACHE, 1)
`__FATORI_MACRO_DEF(FATORI_BRANCH_PRED, 1)
`__FATORI_MACRO_DEF(FATORI_RV32M, ibex_pkg::RV32MSingleCycle)
`__FATORI_MACRO_DEF(FATORI_REGFILE, ibex_pkg::RegFileFPGA)
\end{lstlisting}

The Py2HWSW Layer contains the \ac{RTL} implementation of the system. Every hardware modification made during this work is implemented in this layer, starting from an existing \ac{RISC-V} baseline selected through the process described in \cref{chap2:analysis} to the implementations described in \cref{chap4:el2,chap5:ibex,chap6:fatori_hardware}. It is configured through 19 interface files, including 6 \texttt{.svh} headers, 5 \texttt{.h} headers, and 8 \texttt{.tcl} scripts. The \ac{Fatori-V} Layer is responsible for generating those 19 files and placing them in the correct locations inside the hardware tree. It also provides the file-based \ac{UI}, integrates the \ac{FI} tool, implements run and session automation, and organizes the metric data of the underlying layer into tables.

The \ac{Fatori-V} repository is organized by subsystems, as shown in \cref{fig:fatori_repo_tree}.

\vspace{-10pt}

  \end{center}
  \caption{Fatori-V repository hierarchy.}

The user interacts mainly with \texttt{runs/}, \texttt{inputs/}, \texttt{config/}, and \texttt{results/}. The \texttt{runs/} folder stores experiment \texttt{.yaml} files, one per experiment, and for each run file the platform creates a dedicated output folder under \texttt{results/}. The folders \texttt{inputs/} and \texttt{config/}, together with \texttt{fatori\_settings.py}, define the global configuration of the system. A more detailed view of the user-facing folders is given in \cref{chap3:UI}.

The \texttt{architecture/} folder contains the Py2HWSW layer, later described in \cref{chap5:iob-soc-ibex}. The \texttt{fi/} folder contains the fault injection tool developed in this thesis. Despite being a major feature of \ac{Fatori-V}, it is implemented as a separate tool and can be integrated into other systems, and it is described in \cref{chap8:fatori_fi}.

The \texttt{benchmarks/} folder wraps the integrated benchmarks. It includes clones of upstream benchmark repositories, Makefiles that extract and build their sources, and local port files that adapt them to this platform. The main entry point for \ac{Fatori-V}'s implementation is \texttt{fatori-v.py}, where the Run/Session program is implemented, and \texttt{scripts/} contains the supporting Python code organized by areas (e.g., \texttt{cli/}, \texttt{logging/}, \texttt{reports/}, or \texttt{pblocks/}).

\section{User Interface and Generated Outputs}
\label{chap3:UI}

\ac{Fatori-V} is controlled through a file-based \ac{UI}. The interface is divided into run files, global configuration files, and result directories. Run files define individual experiments. Global files setup the platform and provide external inputs. Result directories store the generated files, logs, reports, and final tables produced by each run.

Each experiment is described by one run \texttt{.yaml} file placed under \texttt{runs/}. The run file selects the target board, architectural features, \acp{FTM}, benchmark sessions, optional \ac{FI} campaign, and reported metrics. A complete example is provided in \cref{app_a:yaml}. This interface keeps experiment setup at the configuration level, so the user can define several runs without editing the hardware or software implementation directly. A run file is organized into three top-level blocks. The \texttt{run} block defines identification and execution context. The \texttt{general} block defines high-level switches, including hardware features, benchmarks, \ac{FI}, and metric level. The \texttt{specifics} block defines detailed parameters for the features enabled in \texttt{general}. \Cref{tab:yaml_run_fields,tab:yaml_general_fields,tab:yaml_specifics_fields} summarize the most relevant fields.

Global files, summarized in \cref{fig:ui_inputs_config}, are configured once and reused across experiments. The \texttt{inputs/} folder stores hardware, software, and TCL files that are not generated, and are instead copied to the build tree. The destination of each copied file is defined through the corresponding \texttt{locations.yaml} file. \texttt{fatori\_registers.yaml} assigns stable IDs and names to wrapped registers. \texttt{gen\_locations.yaml} maps generated files to their destinations. \texttt{system\_hierarchy.yaml} defines parent-child relations between hardware modules for mechanisms that require hierarchical targeting. The \texttt{config/} folder configures the platform itself. The \texttt{validation\_checks.py} file defines the rules used to validate and normalize the run file, as described in \cref{chap7:software_overview}. \texttt{metrics\_definitions.py} defines measured and computed metrics, including the formulas needed to compute them. \texttt{log\_levels.py} and \texttt{messages\_formats.py} define the event-based centralized logging system. \texttt{fatori\_settings.py} stores defaults such as board selection, clock, baud rate, and dry-run mode. Selected settings can also be overridden through the \ac{CLI}.

Each run creates one output directory under \texttt{results/\textless run name\textgreater/}, whose structure is shown in \cref{fig:ui_results_layout}. The top level stores the original run file, the verified run file produced by validation, and the consolidated \texttt{CSV} and \texttt{XLSX} result tables. The \texttt{gen/} subdirectory mirrors the files generated for that run, including \texttt{.svh}, \texttt{.h}, \texttt{.tcl}, and pblock-related files. The \texttt{reports/} subdirectory stores build reports and parsed implementation metrics. The \texttt{sessions/} subdirectory stores one folder per benchmark session, including benchmark logs, extracted benchmark metrics, and \ac{FI} logs when injection is enabled. The file \texttt{sessions/bench\_metrics.csv} aggregates per-session benchmark metrics before they are merged with implementation data. The final outputs of a run are the consolidated \texttt{CSV} and \texttt{XLSX} result tables, summarized in \cref{tab:results_table_benchmark_excerpt,tab:results_table_fpga_excerpt}, that combine benchmark metrics extracted from \texttt{sessions/bench\_metrics.csv} with \ac{FPGA} implementation metrics parsed from Vivado reports into \texttt{\textless board name\textgreater\_metrics.csv}.
